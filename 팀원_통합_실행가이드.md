# 파트2·3 통합 실행 가이드 (팀원용, 처음부터)

> 드론 카메라로 목표(갯끈풀)를 찍으면 → 로봇이 장애물을 피해 그 앞까지 자율주행하는 시뮬레이션.
> 이 문서 하나만 순서대로 따라 하면 다른 우분투 PC에서도 똑같이 돌아갑니다.
> ROS를 처음 써도 되도록 "무슨 명령이 뭘 하는지"까지 다 적었습니다.

---

## 0. 전체 그림 — 무엇을 만드는가

파트가 이렇게 물려 있습니다.

```
[카메라 영상 클릭]          드론이 본 목표를 픽셀로 지정        ← 나중에 파트1 AI 탐지로 교체
      │  /detection/pixel (픽셀 u,v)
      ▼
[obstacle_locator]  파트2   픽셀 → 지면 좌표(거리·방위) 계산
      │  /obstacle/ground_point (map 기준 x,y)
      ▼
[goal_relay]        통합    좌표 → Nav2 목표(/goal_pose)로 변환 (+목표 앞에서 멈추게)
      │  /goal_pose (map 기준 목표 자세)
      ▼
[Nav2]              파트3   라이다로 장애물 피하며 목표까지 자율주행
      │  /cmd_vel (속도 명령)
      ▼
   로봇(TurtleBot3) 이동
```

핵심 용어(몰라도 실행은 되지만 알면 편함):
- **노드(node)**: 하나의 실행 프로그램. 각 터미널에서 하나씩 켭니다.
- **토픽(topic)**: 노드끼리 데이터를 주고받는 통로 (`/goal_pose` 처럼 슬래시로 시작).
- **TF**: 좌표계 간의 관계(map↔로봇↔카메라)를 알려주는 시스템. 이게 있어야 "로봇이 map 어디에 있는지" 계산됩니다.
- **Gazebo**: 3D 물리 시뮬레이터(가상 세계). **Nav2**: 자율주행 소프트웨어.

## 1. 사전 준비 (환경)

- **Ubuntu 22.04** + **ROS2 Humble** 이 이미 설치돼 있어야 합니다.
  (아직이면 ROS2 Humble 공식 설치 먼저: https://docs.ros.org/en/humble/Installation.html — 반드시 22.04/Humble 조합. 24.04 아님)
- 인터넷 연결 + `sudo` 권한.

설치됐는지 확인:
```bash
printenv ROS_DISTRO        # humble 이 떠야 함
```

## 2. 필요한 패키지 설치 (한 번만)

터미널에서 아래를 한 번 실행합니다. (각 줄이 무엇인지 주석 참고)

```bash
sudo apt update
sudo apt install -y \
  ros-humble-navigation2 ros-humble-nav2-bringup \  # 자율주행(Nav2) 본체
  ros-humble-turtlebot3-gazebo ros-humble-turtlebot3-msgs \  # 로봇 모델 + Gazebo 연동
  ros-humble-gazebo-ros-pkgs \                       # Gazebo ↔ ROS 다리
  ros-humble-cv-bridge python3-opencv \              # 카메라 영상 처리(클릭 창)
  git                                                # 코드 내려받기
```

로봇 모델을 매번 지정하도록 설정(한 번만):
```bash
echo 'export TURTLEBOT3_MODEL=waffle' >> ~/.bashrc
source ~/.bashrc
```
> `waffle`은 라이다 + 카메라가 달린 TurtleBot3 모델입니다. 이 프로젝트는 waffle을 씁니다.

설치 확인:
```bash
ros2 pkg prefix nav2_bringup     # 경로가 뜨면 Nav2 설치 성공
```

## 4. 파일별 역할 (무엇이 무엇을 하는가)

```
drone_project/
├─ launch/
│   └─ spartina_nav2.launch.py     ★ 무대 세팅: Gazebo + 로봇 소환 + 좌표계(TF)
├─ worlds/
│   └─ spartina_world.world           가상 세계: 바닥 + 드론카메라 + 목표기둥 + 장애물3개
├─ config/
│   └─ nav2_explore.yaml               Nav2 설정(코스트맵·경로계획·제어). 자율주행의 두뇌
└─ scripts/
    ├─ obstacle_locator_node.py    ★ [파트2] 픽셀 → 지면 좌표 계산 (핵심 알고리즘)
    ├─ goal_relay_node.py          ★ [통합]  파트2 좌표 → Nav2 목표로 변환하는 "다리"
    ├─ click_pixel_node.py            [파트2] 카메라 영상 창 띄우고 클릭 → 픽셀 발행
    ├─ target_follower_node.py        [파트2] 단순 추종(Nav2 쓰면 안 씀, 참고용)
    ├─ pixel_to_ground.py             [파트2] 투영 수식 (obstacle_locator가 내부적으로 사용)
    └─ validate_projection.py         [파트2] 좌표 정확도 검증용(오프라인)
```

### 각 파일 상세

**`launch/spartina_nav2.launch.py` — 무대 세팅**
한 번 실행하면 아래를 동시에 켭니다.
- Gazebo 시뮬레이터 + 갯벌 월드 로드
- TurtleBot3 waffle 로봇을 (-2, 0)에 소환
- **static TF 2개**를 발행:
  - `map → odom` (동일 좌표, identity): Nav2가 로봇 위치를 map 기준으로 알게 함.
  - `map → drone_camera_optical`: 드론 카메라(상공 8m, 수직 하방)의 위치·자세. 파트2 계산의 기준.
- ※ 이 좌표계 통일이 파트2(원래 world 기준)와 파트3(map 기준)를 붙이는 핵심입니다. 전부 `map`으로 맞췄습니다.

**`worlds/spartina_world.world` — 가상 세계**
- 바닥 평면
- **드론 카메라**: (0,0) 상공 8m에서 아래를 봄. 토픽 `/drone/image_raw`, `/drone/camera_info` (640×480).
  - ※ 로봇(waffle) 자체 카메라(`/camera/*`)와 겹치지 않도록 일부러 `/drone/*`로 분리했습니다.
- **목표 기둥**(빨간 원기둥, 갯끈풀 대체): (2, 0). 충돌체 있음 → 라이다가 실제 장애물로 인식.
- **장애물 3개**: (0,0)[경로 한가운데, 회피 시연용], (3,-1.5), (3.5,1.5).

**`config/nav2_explore.yaml` — 자율주행 두뇌**
- 미리 만든 지도 없이(탐험 모드) 라이다 스캔으로 실시간 코스트맵을 만들어 장애물을 피합니다.
- 경로계획(planner), 지역 제어(controller/DWB), 회피 행동 등 Nav2 설정 전체.

**`scripts/obstacle_locator_node.py` — [파트2] 좌표 계산 (핵심)**
- 입력: `/drone/camera_info`(카메라 K행렬), `/detection/pixel`(목표 픽셀)
- 처리: 핀홀 카메라 모델로 픽셀 → 지면(z=0) 좌표를 계산. TF로 카메라 자세를 읽어 정확도 확보.
- 출력: `/obstacle/ground_point`(목표의 map 절대좌표) 등 4개 토픽.
- 실행 시 `world_frame:=map` 을 꼭 줍니다(좌표계를 map으로 통일했기 때문).

**`scripts/goal_relay_node.py` — [통합] 다리**
- 입력: `/obstacle/ground_point` (파트2가 준 목표 좌표)
- 처리: 이걸 Nav2가 이해하는 `/goal_pose`로 변환. 두 가지 스마트 처리:
  1. 목표가 실제로 바뀌었을 때만 한 번 발행(Nav2가 계속 리셋되는 것 방지).
  2. **standoff**: 목표에서 0.8m 앞 지점을 목표로 삼아, 로봇이 기둥에 부딪히지 않고 앞에서 멈추게 함.
- 출력: `/goal_pose`

**`scripts/click_pixel_node.py` — [파트2] 클릭 입력**
- `/drone/image_raw`(드론 영상)를 창에 띄우고, 마우스로 클릭한 픽셀을 `/detection/pixel`로 발행.
- 데모용. 나중에 파트1 AI 탐지가 이 자리를 대체합니다.

## 5. 실행 순서 (터미널 5개)

각 단계는 **새 터미널**에서 실행합니다. 앞 단계가 뜬 걸 확인하고 다음으로 넘어가세요.

> 재실행/문제 시 먼저 정리: 아무 터미널에서
> ```bash
> pkill -9 gzserver; pkill -9 gzclient; pkill -9 rviz2; \
> pkill -f controller_server; pkill -f planner_server; pkill -f bt_navigator; pkill -f lifecycle_manager
> ```

### 터미널 1 — 시뮬레이터 + 로봇 + 좌표계
```bash
export TURTLEBOT3_MODEL=waffle
ros2 launch ~/drone_project/launch/spartina_nav2.launch.py
```
확인: Gazebo 창이 뜨고, 로그에 `Successfully spawned entity [waffle]`, `map to odom`, `map to drone_camera_optical` 가 보이면 성공. 화면엔 로봇·빨간 기둥·회색 상자들이 보입니다.

### 터미널 2 — Nav2 (자율주행 본체)
```bash
export TURTLEBOT3_MODEL=waffle
ros2 launch nav2_bringup navigation_launch.py use_sim_time:=True \
  params_file:=$HOME/drone_project/config/nav2_explore.yaml
```
확인: `Managed nodes are active` 가 뜰 때까지 기다립니다. (이게 떠야 Nav2 준비 완료)

### 터미널 3 — 변환 다리
```bash
python3 ~/drone_project/scripts/goal_relay_node.py
```
확인: `goal_relay_node 시작 ...` 이 뜨고 대기 상태.

### 터미널 4 — 파트2 좌표 계산 (★ world_frame:=map 필수)
```bash
python3 ~/drone_project/scripts/obstacle_locator_node.py --ros-args \
  -p use_tf2:=true -p world_frame:=map \
  -p camera_frame:=drone_camera_optical -p robot_frame:=base_link
```
확인: `픽셀(...) → 지면(X=..., Y=...)` 로그가 계속 나오면 계산 중.

### 터미널 5 — 카메라 영상 클릭
```bash
python3 ~/drone_project/scripts/click_pixel_node.py
```
확인: **"Drone Camera - click target"** 창이 뜨고, **위에서 내려다본 탑뷰**(기둥·상자가 윗면으로 보임)면 정상.

## 6. 데모 시연

카메라 창(터미널 5)에서:
1. **아래쪽 빈 곳을 클릭** → 로봇이 출발점 쪽으로 이동(로봇을 목표에서 멀리 둠).
2. 로봇이 멈추면 → **빨간 기둥을 클릭**.
3. 로봇이 (0,0) 상자를 **옆으로 우회**하며 접근 → 기둥을 라이다로 인식해 **약 0.8m 앞에서 멈추고 기둥을 바라봄**.

이때 터미널 4 로그의 `지면(X,Y)`가 기둥 실제 위치(2, 0) 근처면 좌표 계산이 정확한 것입니다.

### 멈추는 거리(standoff) 조절하기

기둥에서 몇 m 앞에 멈출지는 `standoff` 값으로 정합니다. 기본값은 0.8m입니다.

1. **터미널 3**(goal_relay가 돌아가는 창)에서 `Ctrl + C` 로 노드를 끕니다.
2. 같은 터미널에서 뒤에 옵션을 붙여 다시 실행합니다.

```bash
python3 ~/drone_project/scripts/goal_relay_node.py --ros-args -p standoff:=0.6
```

- `--ros-args -p 이름:=값` 은 ROS2 노드에 설정값을 넘기는 문법입니다
  (obstacle_locator 실행 시 쓰는 `-p use_tf2:=true` 와 같은 방식).
- 시작 로그에 `standoff=0.6` 이 찍히면 적용된 것입니다.
- ⚠️ 0.5 이하로 너무 작게 주면, 목표가 기둥 충돌체 + Nav2 안전여유(팽창) 영역 안으로 들어가
  로봇이 "도착"으로 인정하지 못하고 계속 맴돌 수 있습니다. **0.6~0.9 사이**를 권장합니다.
- 바꾼 효과를 보려면 로봇을 목표에서 멀리 보냈다가(빈 곳 클릭) 다시 기둥을 클릭하세요.

> 화면 녹화로 이 과정을 찍으면 멘토 증빙 영상이 됩니다.

## 7. 목표 직접 주기 (클릭 없이 테스트)

카메라 없이 파이프라인만 확인하려면, 좌표를 직접 넣어도 됩니다.
```bash
# 파트2를 흉내: 목표 좌표를 직접 발행 (기둥 위치)
ros2 topic pub --once /obstacle/ground_point geometry_msgs/msg/Point "{x: 2.0, y: 0.0, z: 0.0}"

# 또는 Nav2에 목표를 완전히 직접(다리 없이)
ros2 topic pub --once /goal_pose geometry_msgs/PoseStamped \
  "{header: {frame_id: 'map'}, pose: {position: {x: 1.3, y: 0.0, z: 0.0}, orientation: {w: 1.0}}}"
```

## 8. 문제 해결 (자주 겪는 것)

| 증상 | 원인 / 해결 |
|---|---|
| `ros2 pkg prefix nav2_bringup` → Package not found | Nav2 미설치 → 2장 설치 명령 실행 |
| 로봇이 안 움직임, `/cmd_vel` 비어있음 | 목표가 장애물 "속"이면 Nav2가 거부. 기둥 정중앙(2,0) 대신 앞쪽(1.3,0)으로. 또는 목표가 안 꽂혔으면(--once 씹힘) 다시 발행 |
| 카메라 창이 옆에서 본 화면(수평선 보임) | 로봇 자체 카메라와 토픽 충돌. 이 저장소는 이미 `/drone/*`로 분리돼 있으니 최신본 `git pull` 확인 |
| `TF_OLD_DATA ... ignoring data from the past` 경고 도배 | Gazebo를 재시작하면 시뮬 시계가 리셋됨. 재시작 전부터 켜둔 `tf2_echo` 등 오래된 조회 노드가 원인 → 그 터미널 Ctrl+C. **Gazebo 재시작 시 Nav2·조회노드도 껐다 켜기** |
| 목표를 줬는데 Nav2가 반응 없음 | `ros2 topic pub` 에서 `--once` 가 타이밍상 씹힐 수 있음 → `--once` 빼고 실행 후 2~3초 뒤 Ctrl+C |
| 로봇이 기둥에 겹침 | 기둥 충돌체 + goal_relay standoff 로 해결됨. 최신본인지 `git pull` 확인 |
| git `LF/CRLF` 경고 | 무해함. 무시 |

### 좌표계·토픽 점검 명령
```bash
ros2 node list                                  # 켜진 노드 확인
ros2 topic list | grep drone                    # /drone/image_raw, /drone/camera_info 있나
ros2 run tf2_ros tf2_echo map base_link         # 로봇이 map 기준으로 잡히나 (좌표 나오면 OK)
ros2 topic echo /obstacle/ground_point --once   # 파트2가 목표 좌표 뱉나
ros2 topic echo /cmd_vel                         # Nav2가 속도 명령 내보내나
```

## 9. 좌표계 한 줄 요약

이 프로젝트는 `map = odom = Gazebo원점` 을 전부 동일(identity)하게 두었습니다.
그래서 **드론이 계산한 좌표 = Gazebo 좌표 = Nav2 목표 좌표** 가 그대로 통합니다.
- `obstacle_locator` 는 반드시 `world_frame:=map` 로 실행.
- `map→odom`, `map→drone_camera_optical` 는 런치가 자동 발행 (별도 static TF 켜지 말 것).
